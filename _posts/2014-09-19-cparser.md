Recently I stumbled across the library [libfirm](http://pp.ipd.kit.edu/firm/) and its associated frontend cparser, which together constitute a working and (they claim) mature C compiler.
Some of my work lately has necessitated inspecting the generated assembly for code I've written.
For those unfamiliar with the inner workings of compilers, they turn things like C code:

{% highlight c %}
void csr_matvec(int m, int *ptr, int *node, double *val,
                double *x, double *y)
{
  int i, j, k;
  double z;

  for (i = 0; i < m; i++) {
    z = 0.0;

    for (k = ptr[i]; k < ptr[i + 1]; k++) {
      j = node[k];
      z += val[k] * x[j];
    }

    y[i] = z;
  }
}
{% endhighlight %}

into assembly code, which looks a little bit like this:

{% highlight asm %}
  movl -8(%rbp), %eax
  cltq
  leaq 0(,%rax,4), %rdx
  movq -56(%rbp), %rax
  addq %rdx, %rax
  movl (%rax), %eax
  movl %eax, -20(%rbp)
  movl -8(%rbp), %eax
  cltq
  leaq 0(,%rax,8), %rdx
  movq -64(%rbp), %rax
  addq %rdx, %rax
  movsd (%rax), %xmm1
  movl -20(%rbp), %eax
  cltq
  leaq 0(,%rax,8), %rdx
  movq -72(%rbp), %rax
  addq %rdx, %rax
  movsd (%rax), %xmm0
  mulsd %xmm1, %xmm0
  movsd -16(%rbp), %xmm1
  addsd %xmm1, %xmm0
  movsd %xmm0, -16(%rbp)
  addl $1, -8(%rbp)
{% endhighlight %}

Assembly code is about one step above the raw binary code that your processor understands.
The variables %rax, %rdx, and so forth are the processor's general purpose registers, which store integers or pointers; the variables %xmm0, %xmm1, ... are the floating point registers.
I haven't shown all the assembly code for the C function above, as it's quite long; you can copy the function above into a file and run

`gcc -S filename.c`

to get the raw assembly code output.

As you might imagine, trying to understand the translation from C code into the very elementary instructions in the generated assembly code is a rather difficult exercise in compiling code in your head.
When compiler optimizations are turned on, the generated assembly code can be quite opaque.
This all makes for a steep learning curve if you're interested in code generation.

Unlike GCC, the cparser compiler has the wonderful feature that all generated assembly code is annotated with its origin in the C source code.
For example, here's the output of cparser for the interior loop of  the C function:

{% highlight asm %}
  movslq %eax, %r15           /* amd64_Movs T[359:53]  */
  movslq (%rdx,%r15,4), %r14  /* amd64_Movs T[361:55] csr.c:10:21 */
  movsd (%r8,%r14,8), %xmm1   /* amd64_xMovs T[368:62] csr.c:11:28 */
  mulsd (%rcx,%r15,8), %xmm1  /* amd64_xMuls T[378:72] csr.c:11:25 */
  addsd %xmm1, %xmm0          /* amd64_xAdds T[380:74] csr.c:11:15 */
  leal 1(%rax), %eax          /* amd64_Lea T[385:79] csr.c:9:43 */
{% endhighlight %}

With this extra information, we can easily connect the floating point operations in the third through fifth lines of this assembly code with the accumulation step on line 11 of the C code.
Beyond the libfirm/cparser website, I've seen very little information about it elsewhere on the internet; let's hope it gets the attention it deserves.
